# 大型复杂系统开发中的一个常见误区：过度里程碑规划 vs Vibe Coding

## 1. 问题背景

在开发复杂系统（尤其是 **模拟系统、Agent 系统、开放世界系统、研究型项目**）时，很多开发者会采用传统的软件工程方式：

制定大型里程碑（Milestones），例如：

* M1：系统架构
* M2：核心功能
* M3：扩展功能
* M4：性能优化

这种方式在 **需求明确、结构稳定的软件项目** 中通常是有效的。

但在 **探索型系统（Exploratory Systems）** 中，这种方法经常会导致严重的问题。

---

# 2. 过度里程碑规划的问题

在探索型系统中，开发者通常会提前设计大量模块，例如：

* World / Environment
* Agent
* Memory
* Economy
* Communication
* Inventory
* Event System
* Planning
* Reflection
* Social Systems

这些模块往往 **相互依赖**。

结果开发流程会变成：

设计架构
→ 实现一部分
→ 发现缺少依赖模块
→ 扩展架构
→ 架构复杂度继续上升

最终形成一种循环：

设计
→ 依赖爆炸
→ 重构
→ 再设计

这种情况通常被称为：

**Architecture Overdesign（架构过度设计）**

常见结果包括：

* 系统难以真正运行
* 开发进度缓慢
* 代码结构混乱
* 项目目标逐渐偏离

---

# 3. 为什么探索型系统更容易出现这个问题

探索型系统通常具有以下特点：

* 系统行为难以提前预测
* 关键设计需要通过实验验证
* 模块边界在早期阶段并不清晰
* 需求会随着实验不断变化

典型例子包括：

* Agent 模拟
* 复杂社会模拟
* 经济模拟
* AI 驱动世界
* 研究型 AI 项目
* 新型产品原型

在这些系统中，**真正有效的架构通常是“运行出来的”，而不是“设计出来的”。**

---

# 4. Vibe Coding 方法

一种更适合探索型系统的开发方式是所谓的 **Vibe Coding**（一种偏实验驱动的开发风格）。

其核心思想是：

**让系统尽快运行，然后通过观察逐步演化结构。**

典型流程是：

实现最小功能
→ 运行系统
→ 观察行为
→ 调整规则
→ 增加模块

而不是：

设计完整架构
→ 一次性实现

---

# 5. 最小核心原则（Minimal Spine）

为了避免架构膨胀，可以将系统拆分为两类结构：

## 5.1 Spine（系统脊柱）

Spine 是系统的 **最小不可替代核心**。

它应该满足两个条件：

* 数量极少
* 长期稳定

例如在许多模拟系统中，核心可能只包含：

* Environment / World
* Agent
* State / Memory
* Actions

只要这些存在，系统就可以持续运行。

---

## 5.2 Modules（模块）

其它功能都应该作为 **可插拔模块**：

* Economy
* Trade
* Politics
* Communication
* Inventory
* Planning
* Social Systems

模块的特点：

* 可以独立开发
* 可以随时加入
* 可以随时移除
* 不影响核心运行

---

# 6. 一个实用判断标准

在设计新系统模块时，可以问一个问题：

**如果不实现这个模块，系统还能运行吗？**

如果答案是：

可以运行

那么这个功能应该被设计为 **模块（Module）**，而不是核心架构。

---

# 7. 渐进式系统演化

复杂系统更适合采用 **渐进式演化路径**：

### Stage 1：最小运行系统

只实现核心循环，使系统能够运行。

### Stage 2：行为复杂度

增加动机、关系或规则，使行为变得更丰富。

### Stage 3：资源与约束

引入资源系统或环境限制。

### Stage 4：高层结构

在系统稳定后再加入：

* 经济
* 政治
* 社会结构
* 长期规划

---

# 8. 历史经验

许多成功的复杂模拟项目都是通过 **简单核心 + 长期演化** 发展起来的，例如：

* Dwarf Fortress
* The Sims
* RimWorld
* Minecraft

它们的早期版本都非常简单，但通过持续运行和迭代逐渐形成复杂系统。

---

# 9. 关键结论

对于探索型复杂系统：

**过度规划架构通常会阻碍开发。**

更有效的方法是：

* 建立最小核心（Minimal Spine）
* 快速运行系统
* 通过实验观察行为
* 使用模块逐步扩展

目标不是一次性设计完整系统，而是：

**让系统先活起来，然后逐渐演化。**
