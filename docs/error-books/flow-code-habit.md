# 错题本 — 代码修改习惯

> 适用场景：改代码、改文档、修 P0/P1、批量操作

### 记录规则

- **条目结构**：❌ 错误做法 + ✅ 正确做法，整条控制在 **5 行以内**
- **❌/✅ 写法**：只写核心规则，禁止多层加粗嵌套（加粗最多一层）
- **`>` 注释**：只放归因和 postmortem 链接，禁止复述场景
- **复犯**：只追加频率标记（🟡×N / 🔴×N）
- **归属**：条目只写本范围内容，跨范围写到对应文件

### DEV-6 改代码不 grep 引用 + 不复用 pattern `🟡中频×3`

❌ 改签名不 grep 全量引用→下游断裂；不 grep 同类实现从零写；只改主调用链，遗漏次要入口
✅ 改前 grep 同类 pattern 复用→grep 全量引用→改后对照设计文档→涟漪推演。新增函数：先 Read 同文件已有函数对齐事务模式/错误格式/命名

### DEV-24 更新文档只改局部不扫全文 `🟡×4`

❌ 只改最明显的一处，其余过时内容全部遗漏；改了主索引（_index.md）但漏扫二级索引（flow-rules.md 等）；新增规则后未同步更新相关流程文档
✅ 更新文档前先全文扫描列出所有过时点，一次性全部更新。涉及索引体系时，主索引和所有二级索引必须同步扫描。**新规则落地时必须同步 grep 所有相关流程文档，确认无冲突**
> 归因 B：PR #26 CR 发现 flow-rules.md 条目归属未同步；PR #28 CR 发现 DEV-76 新规则未同步更新 worktree-workflow.md

### DEV-77 文档引用外部链接未验证有效性 `🟢`

❌ 写 README 时直接写 Releases/外部链接，未验证链接是否真实存在，导致死链上线
✅ 写文档前凡引用外部链接必须先验证：`gh release list`（Releases）/ `gh api repos/<owner>/<repo>`（仓库资源）/ WebFetch 实际访问确认
> 归因 A：禁止脑补规则有，但写 README 时未将"外部链接"纳入需要验证的范围。注意：禁止用 curl/wget 验证，走 gh CLI 或 WebFetch

### DEV-78 流程文档 shell 命令未标注执行目录 `🟢`

❌ 流程文档里的 shell 命令没有说明在哪个目录执行，读者按字面在 worktree 里执行 `git worktree remove` 报 Permission denied
✅ 流程文档每个 shell 代码块，凡涉及目录敏感操作必须在注释里标明执行位置（如 `# 在主仓库目录执行`）
> 归因 C：新场景，现有规则未覆盖"流程文档命令执行位置"

### DEV-80 实现前未横向对比同类项目方案 `🟢`

❌ 上来就读自己的代码、顺着现有实现优化，没有先搜同类项目怎么做，导致方向错误（如用 readPixels 做碰撞检测，而业界全部用几何 hitTest）
✅ 遇到"交互检测/渲染/协议"类问题，先搜 2~3 个同类开源项目的实现，确认方向正确后再动手
> 归因 C：新场景，现有规则未覆盖"动手前横向对比"步骤

### DEV-29 P0/P1 修复列表漏项 + 执行碎片化 `🟢`

❌ CR 列了 5 个 P1 只修 4 个；独立修改逐个串行
✅ P0/P1 逐条核销不漏项；独立修改并行发出；独立文件更新并行

### DEV-47 批量/seed 幂等设计未考虑"部分成功" `🟢`

❌ 照搬 `count > 0` 跳过的幂等模式到有外部 API 依赖的场景，部分失败后永远无法补全
✅ 涉及外部依赖的批量操作用细粒度差集（按业务键比对），每条用 savepoint 包裹，单条失败不影响整体

### DEV-63 Review 评论不贴合现状时未做等价落地 `🟢`

❌ 只修字面上最容易对齐的 review 评论；评论和当前实现不完全贴合时，既不转译底层要求，也不明确驳回，结果漏掉真正要修的问题
✅ review 必须逐条核销。对不完全贴合现状的评论，先提炼底层要求，再选择等价修复或给出明确驳回理由，直到评论线程闭环

### DEV-69 文档中硬编码环境相关值（分支名、路径格式） `🟢`

❌ 文档示例中硬编码 `master`、Windows 反斜杠路径等环境相关值，换环境即失效
✅ 用占位符 `<主分支>` 加注释说明实际值，bash 代码块中路径统一用正斜杠
> 归因 C：新场景，无对应检查项。PR #17 CR 发现

### DEV-72 新功能前端无条件启动，未与后端配置协商 `🟢`

❌ 前端在连接建立后无条件启动截屏，后端 `screen.enabled=false` 时仍截屏发送，开关形同虚设
✅ 前端功能启动必须与后端配置协商：通过 health/capability API 获取开关和参数，后端说关就不启动
> 归因：只在后端做了开关，假设前端会"自然"配合，没考虑前端是独立进程需要显式协商。PR #23 CR 发现

### DEV-81 在错误链路上叠补丁不回退找根因（补丁螺旋）`🟢`

❌ 发现 A 不工作→补 B→B 引入新问题→补 C→循环。2 个分支 18 轮补丁，每次 commit 写"根治/彻底修复"但只修表面症状，底层平台限制从未被识别
✅ 连续 2 次补丁未解决时强制停下：①最小空白窗口验证平台行为 ②搜索平台已知 bug/限制 ③列出链路所有副作用 ④回退到最后正常状态换完全不同的链路
> 归因 C：Electron transparent 窗口 setPosition 触发假 resize 是 DWM 层限制，18 轮应用层补丁全部无效。正解是 Win32 WM_SYSCOMMAND SC_MOVE 一行调用。根本原因：未先验证平台限制就动手，每层补丁遮蔽上层问题

### DEV-82 平台 API 问题未做最小复现就动手修 `🟢`

❌ 在完整业务代码里反复调试平台行为（Electron/Win32/DWM），每次改动夹杂业务逻辑副作用，无法判断是平台问题还是自己的 bug
✅ 涉及平台 API 异常行为时，第一步建最小可独立复现的示例（脱离业务代码单独验证平台行为）。能复现 = 平台限制，不能复现 = 自己的 bug。确认后再回业务代码修
> 归因 C：拖拽 18 轮全在业务代码里调试，从未隔离验证 setPosition 本身是否触发 resize。最小复现 5 分钟就能定位

### DEV-83 同一功能/链路连续 3 次 fix commit 未强制停下 `🟢`

❌ 对同一功能/同一链路/同一文件簇连续提交 3+ 次 fix，每次 commit message 写"修复/根治"但问题依旧，没有客观信号触发回退
✅ 同一功能/同一链路/同一文件簇累计 3 次 fix commit 未解决 = 强制停下。执行：①`git log --oneline` 确认 fix 次数 ②列出每次 fix 改了什么、为什么没解决 ③回退到第一次 fix 前的状态 ④换完全不同的技术路径
> 归因 C：18 轮 commit 中多次写"根治""彻底修复"，但没有客观计数机制触发停下。需要硬性数字门槛

### DEV-84 平台能力边界未确认就写应用层代码 `🟢`

❌ 假设平台 API 按文档描述工作，直接写应用层逻辑。遇到异常后在应用层打补丁绕过，越绕越深
✅ 使用平台 API 前，先确认能力边界：①查官方文档已知限制 ②搜 GitHub Issues 同类问题 ③最小复现验证实际行为。三步都确认后再写应用层代码
> 归因 C：Electron transparent 窗口 + setPosition 的 DWM 限制在 GitHub Issues 里有大量报告，但从未搜索过。直接假设 setPosition 正常工作

### DEV-85 修复结论无证据，滥用"根治/彻底修复" `🟢`

❌ commit message 写"根治""彻底修复""完美解决"但没有验证证据，下一轮又回退。18 轮中至少 5 次写了"根治"
✅ 禁止在 commit message 中使用"根治/彻底修复/完美解决"等词，除非附带验证证据。commit message 只描述改了什么，不做效果承诺。效果验证写在 PR description 或测试日志里
> 归因 C：commit message 的乐观措辞制造虚假进展感，掩盖了"其实没解决"的事实，降低了回退决策的紧迫性
